# 팀원 Done 프로세스

> Agent Teams 모드에서 팀원이 작업 완료 시 수행하는 5단계 프로세스
> `/done` 커맨드의 팀원 버전 (메인 에이전트 전용 Skill 사용 불가 → Bash 직접 호출)

---

## 적용 조건

- Agent Teams 모드로 spawn된 팀원에게만 적용
- 일반 Task 병렬 호출에서는 적용하지 않음
- 팀원 spawn 시 프롬프트에 이 문서 참조가 포함되어야 함

---

## 5단계 프로세스

### 1단계: 변경 분석

```bash
git diff --stat
git diff
```

- 변경된 파일 목록과 내용 확인
- **범위 밖 변경이 있는지 검증** (요청받은 파일만 수정했는지)

### 2단계: 정책 키워드 탐지 → 테스트 전략 결정

변경 내용에서 정책 키워드를 탐지:

| 키워드 | 영향도 | 테스트 필요 여부 |
|--------|--------|-----------------|
| 날짜/기간 계산 로직 | HIGH | 필수 |
| 수치/수량 계산 로직 | HIGH | 필수 |
| 상태 전이 / 상태 머신 | HIGH | 필수 |
| `disabled`, `readonly` 조건부 동작 | MEDIUM | 권장 |
| 필터/정렬/검색 조건 | MEDIUM | 권장 |

- 정책 키워드가 있으면: 정책 보호 테스트 작성 고려
- 순수 퍼블리싱/스타일 변경이면: 테스트 생략 가능

### 3단계: 린트 + 빌드 검증

```bash
{패키지매니저} lint
```

- 린트 에러가 있으면 수정 후 재검증
- 빌드는 팀 리드가 통합 검증 시 수행 (팀원은 린트만)

### 3.5단계: 출시 품질 게이트 요약

- `release-readiness-gate.md` 기준으로 팀원 작업 범위에서 PASS/FAIL을 간단히 정리한다
- FAIL 항목이 있으면 4단계(커밋) 진행 전 수정 또는 팀 리드에 리스크 보고

### 4단계: 커밋

```bash
# 변경된 파일 명시적으로 추가 (git add -A 금지)
git add <변경된 파일들>

# 커밋 (Co-Authored-By 금지)
git commit -m "type: 메시지"
```

**커밋 규칙**: `git-operator.md` 커밋 메시지 규칙을 따른다.
- `Co-Authored-By` 절대 포함하지 않음
- 팀원 작업은 변경 범위가 작으므로 제목만으로 충분 (본문 선택)

### 5단계: 팀 리드에게 보고

SendMessage로 아래 형식의 보고서 전송:

```
## 작업 완료 보고

### 변경 파일
- file1.tsx: 변경 내용 요약
- file2.tsx: 변경 내용 요약

### 변경 요약
{무엇을 왜 변경했는지 1-2줄}

### 검증 결과
- [ ] 린트 통과
- [ ] 범위 밖 변경 없음
- [ ] 커밋 완료

### 주의사항 (해당 시)
{팀 리드가 알아야 할 사항}
```

---

## 금지 사항

- `git add -A` 또는 `git add .` 사용 금지
- `Co-Authored-By` 커밋 메시지에 포함 금지
- `/start`, `/done` 등 Skill 도구 사용 금지 (메인 에이전트 전용)
- 범위 밖 파일 수정 금지
- 디자인 값(variant, font-size 등) 임의 변경 금지
