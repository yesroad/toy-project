# Release Readiness Gate (출시 전 품질 게이트)

> 계획 수립부터 커밋/PR까지 동일 기준으로 검증한다.
> 이 문서는 품질 검토 규칙의 단일 원본(SSOT)이다.

---

## 적용 시점

- 작업 시작 직후: **계획 게이트**
- 구현 완료 직후: **구현 게이트**
- 커밋/PR 직전: **출시 게이트**

---

## Gate 1. 계획 품질

- [ ] 계획을 문서화했는가?
- [ ] 계획 자체를 검토했는가?
- [ ] 검토 결과가 타당한지 다시 검토했는가?
- [ ] 계획이 과도하지 않은가? (YAGNI/KISS)
- [ ] 모르는 부분은 구현 전에 사용자에게 질문했는가?

**실패 조건:**

- 목적/범위/완료 조건이 불명확함
- 요구 범위를 넘는 선행 리팩토링이 포함됨

---

## Gate 2. 구현 적합성

- [ ] 구현이 원래 목적과 직접 연결되는가?
- [ ] 기존 정책/비즈니스 규칙을 임의 변경하지 않았는가?
- [ ] 기존 코드와 재사용/통합 가능한 부분을 우선 활용했는가?
- [ ] 매우 큰 함수/파일을 적절히 분리했는가?
- [ ] 구현 과정에서 불필요해진 코드를 정리했는가?

**실패 조건:**

- 요구사항 대비 과구현 또는 미구현
- 동일 책임 코드가 중복 유지됨

---

## Gate 3. 안정성/보안/사이드 이펙트

- [ ] 잠재 버그/크리티컬 이슈를 점검했는가?
- [ ] 보안 문제(비밀정보 노출, 권한, 입력 검증)를 점검했는가?
- [ ] 사이드 이펙트(다른 화면/도메인 영향)를 점검했는가?
- [ ] 개선한 내용이 기존 동작을 깨지 않는지 확인했는가?

**실패 조건:**

- 치명적 결함 또는 보안 이슈가 열린 상태
- 영향 범위를 설명할 수 없음

---

## Gate 4. 사용자 흐름/운영 품질

- [ ] 실제 사용자 흐름에서 막히는 지점이 없는가?
- [ ] 에러/빈 상태/로딩 상태에서 UX가 유지되는가?
- [ ] 코드 품질(가독성, 책임 분리, 테스트 가능성)이 충분한가?
- [ ] 전체 변경 사항을 다시 한 번 종합 검토했는가?

**실패 조건:**

- 주요 사용자 경로에서 회귀(regression) 발견
- 코드 품질 저하로 유지보수 리스크 증가

---

## Gate 5. 커밋/PR 준비

- [ ] lint/build(필요 시 테스트) 결과가 명확한가?
- [ ] 커밋 범위가 해당 작업으로 한정되는가?
- [ ] PR 설명에 목적/변경점/검증/리스크가 포함되는가?
- [ ] "지금 배포 가능한가?" 질문에 근거 기반으로 답할 수 있는가?

**원칙:**

- 게이트 미통과 상태에서 커밋/PR 진행 금지
- 미해결 리스크는 숨기지 말고 PR 본문에 명시

---

## 출력 템플릿

```markdown
## Release Readiness Gate

### Gate 1. 계획 품질
- 결과: PASS | FAIL
- 근거: ...

### Gate 2. 구현 적합성
- 결과: PASS | FAIL
- 근거: ...

### Gate 3. 안정성/보안/사이드 이펙트
- 결과: PASS | FAIL
- 근거: ...

### Gate 4. 사용자 흐름/운영 품질
- 결과: PASS | FAIL
- 근거: ...

### Gate 5. 커밋/PR 준비
- 결과: PASS | FAIL
- 근거: ...

최종 판정: READY | NOT READY
```
